# Diskless Kafka

A lightweight, diskless Kafka broker implementation in Python that uses **MinIO** (S3-compatible object storage) as its primary and only storage layer. 

By removing local disk persistence, this broker acts as a pure stateless protocol gateway—translating Kafka wire-protocol requests directly into S3 API calls. It natively handles standard Kafka clients (like `kafka-python`) without them knowing the broker has no local disk!

## Core Concepts

1. **Pure Byte Relay:** The broker never deserializes or parses individual Kafka messages. When a producer sends a `RecordBatch` (the standard Kafka batch format), the broker extracts the raw bytes of the batch and writes them verbatim as a single object to MinIO.
2. **End-to-End CRC32C:** Because the `RecordBatch` bytes are never mutated, the CRC32C checksums generated by the producer are perfectly preserved and validated by the consumer.
3. **Log Order via Lexicographic Sort:** MinIO objects are stored under the key format `{topic}/{partition}/{base_offset:020d}.batch`. Because the offsets are zero-padded to 20 digits, standard S3 bucket listing automatically returns batches in perfect chronological log order.
4. **Stateless High-Watermark Recovery:** The broker tracks the partition's next available offset (High Watermark) in memory. If the broker crashes or restarts, it seamlessly recovers the state by scanning the MinIO bucket and reading the header of the last batch.

## Implemented APIs

We support the minimum subset of the Kafka wire protocol required to allow a client to discover the broker, produce messages, and consume them:

- **ApiVersions (API Key 18):** Used during client connection. We selectively advertise support for newer versions of `Metadata` (v5) and `Produce` (v7) so the client negotiates the modern `RecordBatch` format (magic byte 2) rather than the deprecated `MessageSet` format.
- **Metadata (API Key 3):** Returns a hardcoded single-node cluster state. For any topic requested, it claims `localhost:9092` is the undisputed leader of partition `0`.
- **Produce (API Key 0):** Parses incoming message batches, patches the header offset, writes the exact bytes to MinIO, and returns the assigned base offset.
- **ListOffsets (API Key 2):** Allows consumers to find the start or end of the log. Crucial for handling `auto_offset_reset='earliest'`.
- **Fetch (API Key 1):** Scans the MinIO bucket for the `RecordBatch` containing the requested offset, patches the `base_offset` in the header (which falls safely outside the CRC32C checksum window), and streams the bytes back to the consumer.

## Architecture

```text
┌─────────────────┐       Kafka Protocol       ┌─────────────────────┐
│                 │      (TCP / Port 9092)     │                     │
│  Kafka Client   ├───────────────────────────►│   Diskless Broker   │
│ (Producer/Cons) │                            │    (Stateless)      │
│                 │◄───────────────────────────┤                     │
└─────────────────┘                            └──────┬───────▲──────┘
                                                      │       │ S3 API
                                             Write    │       │ Read
                                           RecordBatch│       │ RecordBatch
                                                      │       │
                                               ┌──────▼───────┴──────┐
                                               │                     │
                                               │        MinIO        │
                                               │   (Object Store)    │
                                               └─────────────────────┘
```

## Running the Project

### 1. Start MinIO
The broker requires a local MinIO instance running on `localhost:9000` with the default `minioadmin:minioadmin` credentials.
```bash
# Start a standalone MinIO server
minio server ./data --console-address :9001
```

### 2. Start the Broker
Run the pure Python server. It will automatically create the `diskless-kafka` bucket in MinIO on startup.
```bash
python3 server.py
```

### 3. Test End-to-End
You can run the included test script which spawns a `KafkaProducer`, sends a test message, and then uses a `KafkaConsumer` to fetch it back.
```bash
python3 test_fetch.py
```

## Future Extensions
- Support for multiple partitions and dynamic leader election.
- Paging and chunking for Fetch requests spanning multiple large RecordBatches.
- Long-polling for Fetch requests (currently it immediately returns an empty payload if the consumer is at the tip of the log).
- Implementing the `FindCoordinator` and `JoinGroup` APIs to support consumer groups.
